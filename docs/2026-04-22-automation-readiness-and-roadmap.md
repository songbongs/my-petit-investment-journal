# SSMK 자동화 준비도와 발전 로드맵

작성일: 2026-04-22

## 1. 이 문서의 목적

이 문서는 SSMK 투자 관찰노트를 자동화할 때 무엇을 먼저 자동화하고, 무엇을 조심해야 하며, 어떤 순서로 발전시키면 좋은지 정리하기 위한 문서다.

현재 구현 기준 메모:

```text
이 문서는 장기 로드맵 중심 문서다.
2026-04-23 기준 현재 기본 구현은 Weekly Lab Google Docs 초안 준비,
로그 기록, 에이전트 리뷰 보드, QA 기록까지다.
이메일 발송과 예약 자동화 변경은 아직 기본 흐름에 연결하지 않았다.
```

현재 방향의 핵심 결론은 다음과 같다.

```text
AI가 처음부터 데이터 수집, 점수 계산, 가설 생성, 초보자용 설명, 리포트 초안, 지난 가설 복기 초안까지 맡는다.
사람은 투자 전문가 역할이 아니라 첫 번째 학습자이자 편집장으로 참여한다.
외부 이메일 발송은 사람이 승인한 뒤 진행한다.
```

즉, 이 프로젝트의 자동화 목표는 단순히 시간을 아끼는 것이 아니다. 더 중요한 목표는 사용자가 시장을 배우는 과정을 돕는 것이다.

AI가 할 일:

- 데이터를 모은다.
- 점수를 계산한다.
- 학습용 가설을 만든다.
- 그 가설의 근거 지표를 설명한다.
- 왜 그 지표를 보고 그런 예상이 나왔는지 설명한다.
- 현재 추론의 한계를 솔직하게 적는다.
- 다음에 확인할 항목을 남긴다.
- 지난 가설과 실제 결과를 비교한다.

사람이 할 일:

- AI가 만든 설명을 읽고 배운다.
- 이해되지 않는 지표나 가설을 질문한다.
- 문장이 투자 추천처럼 읽히는지 확인한다.
- 외부 이메일 발송 여부를 승인한다.
- AI가 제안한 자동화 발전 단계나 중요한 변경을 승인/보류/거절한다.

## 2. 자동 데이터 수집이 쉽게 깨지는 이유

자동화는 편리하지만, 투자 데이터 자동 수집은 생각보다 쉽게 깨진다. 초보자용 비유로 말하면, 여러 식당에서 매주 같은 메뉴판을 가져오는 일과 비슷하다. 그런데 어떤 식당은 메뉴판 위치를 바꾸고, 어떤 식당은 이름을 바꾸고, 어떤 식당은 임시 휴업을 한다. 사람은 보고 이해하지만, 자동화 스크립트는 정해진 위치에 정해진 글자가 없으면 실패할 수 있다.

### 2-1. 웹사이트 구조가 바뀐다

예를 들어 어떤 사이트에서 배당률을 가져오도록 자동화를 만들었다고 하자.

처음에는 배당률이 아래처럼 있었다.

```text
Dividend Yield: 3.2%
```

그런데 사이트가 개편되면서 아래처럼 바뀌면 자동화가 못 찾을 수 있다.

```text
Forward Yield
3.2%
```

사람은 둘 다 배당률 관련 정보라고 이해하지만, 자동화 스크립트는 "Dividend Yield"라는 글자를 찾다가 실패할 수 있다.

### 2-2. 같은 이름의 데이터도 의미가 다를 수 있다

예를 들어 배당률 하나만 봐도 기준이 여러 개다.

| 이름 | 의미 | 초보자용 해석 |
|---|---|---|
| Trailing dividend yield | 지난 12개월 실제 배당 기준 | 이미 지급된 배당으로 계산한 배당률 |
| Forward dividend yield | 앞으로 예상되는 배당 기준 | 앞으로도 현재 배당이 유지된다고 보고 계산한 배당률 |
| Indicated annual dividend | 현재 배당이 계속된다고 가정한 연간 배당 | 최근 배당을 1년치로 환산한 값 |

모두 "배당률"처럼 보이지만 실제 의미가 다르다. AI가 이 차이를 모르면 잘못된 가설을 만들 수 있다. 따라서 리포트에는 단순히 숫자만 쓰지 말고, 어떤 기준의 숫자인지 함께 적어야 한다.

### 2-3. 실적 발표일과 이벤트 일정은 바뀔 수 있다

실적 발표일, 배당락일, FDA 일정, 제품 발표 일정은 변경될 수 있다. 특히 "예상 일정"과 "확정 일정"을 구분해야 한다.

예시:

```text
예상 실적 발표일: 2026-04-28
확정 실적 발표일: 2026-04-30
```

AI가 예상 일정을 확정 일정처럼 리포트에 넣으면 신뢰도가 떨어진다. 그래서 일정 데이터에는 `확정`, `예상`, `출처 확인 필요` 같은 상태값이 필요하다.

### 2-4. ADR/OTC 종목은 데이터가 불안정할 수 있다

`LVMUY`처럼 ADR/OTC 종목은 미국 대형 상장주보다 무료 데이터가 덜 깔끔할 수 있다. 가격, 배당, 거래량, 공시 연결이 출처마다 다르게 나올 수 있다.

이런 종목은 자동화에서 제외할 필요는 없지만, `data_confidence = 낮음` 또는 `uncertainty_level = 높음`으로 표시하고, 리포트에서 한계를 분명히 적어야 한다.

### 2-5. 뉴스 자동 요약은 과장될 수 있다

뉴스 제목은 자극적일 수 있다.

나쁜 자동 요약:

```text
회사가 AI 시장을 장악할 가능성이 커졌다.
```

SSMK에 맞는 관찰 기록:

```text
AI 관련 뉴스가 있었지만, 실제 매출과 마진에 반영되는지는 다음 실적에서 확인해야 한다.
```

뉴스는 "분위기"를 만들 수 있지만, 투자 공부에서는 숫자로 확인되는지까지 봐야 한다. 그래서 AI는 뉴스 요약을 할 때 반드시 "이 뉴스가 어떤 지표로 확인되어야 하는지"를 함께 적어야 한다.

## 3. 왜 4주 AI 자동 초안 운영이 필요한가

과거 방향은 "먼저 사람이 직접 운영하고 나중에 자동화"에 가까웠다. 하지만 현재 프로젝트의 목적은 다르다. 사용자가 초보자이고, 이 관찰노트 자체가 사용자의 학습 도구이므로, 처음부터 AI가 분석과 설명을 적극적으로 도와야 한다.

따라서 현재의 4주 운영은 `AI 자동 초안 운영`으로 정의한다.

4주 동안 확인할 것은 다음이다.

```text
AI가 만든 가설이 초보자에게 이해되는가?
근거 지표가 실제로 가설과 연결되는가?
AI가 한계를 솔직하게 말하는가?
지난 가설이 실제 시장, 실적, 뉴스 흐름과 얼마나 비슷했는가?
어떤 데이터 출처가 자주 깨지는가?
```

이 단계는 자동화를 늦추는 단계가 아니다. 오히려 "자동화가 어떤 방식으로 똑똑해져야 하는지"를 배우는 단계다.

## 4. 4주 AI 자동 초안 운영 계획

### 1주차: 작은 범위로 가설 구조 검증

목표:

- 5~7개 종목으로 AI 가설 구조가 읽히는지 확인한다.
- `가설 요약 + 근거 지표/데이터 + 왜 그렇게 해석했나 + 초보자 레슨 + 한계와 다음 확인` 구조를 리포트에 넣는다.
- 사용자가 읽었을 때 "아, 이 지표는 이런 걸 보는 지표구나"라고 이해되는지 확인한다.

운영 범위:

| 항목 | 기준 |
|---|---|
| 종목 수 | 5~7개 |
| 산업 수 | 3개 정도 |
| 리포트 길이 | 짧게 |
| AI 역할 | 점수 계산, 가설 생성, 초보자 설명, 한계 작성 |
| 사람 역할 | 질문, 이해 확인, 발송 승인 |

성공 기준:

```text
사용자가 리포트를 읽고 최소 3개 지표의 의미를 새로 이해하면 성공.
```

### 2주차: 종목 수 확장과 데이터 신뢰도 확인

목표:

- 종목 수를 10~12개로 늘린다.
- 점수, 관찰 등급, 데이터 신뢰도, 불확실성을 함께 본다.
- AI가 어떤 데이터는 자신 있게 쓰고, 어떤 데이터는 한계를 붙이는지 확인한다.

운영 범위:

| 항목 | 기준 |
|---|---|
| 종목 수 | 10~12개 |
| 산업 수 | 5~7개 |
| 필수 항목 | 데이터 신뢰도, 불확실성, 리스크 플래그 |
| AI 역할 | 후보 3개 선정, AI 가설 3개 작성, 한계 고지 |

성공 기준:

```text
점수보다 "왜 그렇게 판단했는지"가 먼저 보이면 성공.
```

### 3주차: 지난 가설과 실제 결과 비교

목표:

- 1~2주차에 세운 가설을 실제 주가, 실적, 뉴스 흐름과 비교한다.
- 맞고 틀림보다 "왜 비슷했는지, 왜 어긋났는지"를 기록한다.
- AI가 틀린 가설을 방어하지 않고, 배운 점을 정리하게 한다.

비교 예시:

| 항목 | 예시 |
|---|---|
| 지난 가설 | 클라우드 성장률이 유지되면 MSFT 관찰 점수가 유지될 가능성이 있다 |
| 실제 결과 | 실적에서 Azure 성장률은 유지됐지만, AI 투자 비용 증가로 마진 우려가 커졌다 |
| 배운 점 | 성장률만 보면 부족하고, AI 투자가 비용으로 먼저 반영되는지도 함께 봐야 한다 |
| 다음 개선 | 클라우드 성장률과 영업마진을 같이 추적한다 |

성공 기준:

```text
AI가 "맞았다/틀렸다"가 아니라 "다음에는 어떤 지표를 추가로 봐야 하는지"를 제안하면 성공.
```

### 4주차: 자동화 강도 조정

목표:

- 안정적으로 자동화할 수 있는 항목과 조심해야 할 항목을 나눈다.
- 데이터 수집 자동화, 가설 생성 자동화, 이메일 발송 승인 흐름을 분리한다.
- AI가 자동화 발전 제안서를 만들고, 사용자가 승인한 항목만 다음 단계로 넘긴다.

분류 기준:

| 구분 | 예시 | 운영 방식 |
|---|---|---|
| 바로 자동화 가능 | 총점 계산, 관찰 등급 변환, 지난주 점수 비교 | 자동 |
| 자동화하되 한계 표시 | 무료 데이터 기반 배당률, 실적 예상일, 뉴스 후보 | 자동 + 신뢰도 표시 |
| AI 해석 필수 | 뉴스 의미, 리스크 원인, 가설 생성 | AI 초안 + 한계 표시 |
| 사람 승인 필수 | 외부 이메일 발송, 가족/지인 공유 | 승인 후 발송 |

성공 기준:

```text
AI가 할 일과 사람이 승인할 일을 구분하면서도, 사용자의 학습 흐름이 끊기지 않으면 성공.
AI가 다음 단계로 넘어가도 된다고 판단해도, 사용자 승인 없이 적용하지 않으면 성공.
```

## 5. 현재 아키텍처의 한계

현재 권장 구조는 다음과 같다.

```text
Google Sheets
  ↓
Google Apps Script 또는 Python
  ↓
AI 점수 계산 / 가설 생성 / 초보자 설명
  ↓
Markdown 또는 Google Docs 리포트 초안
  ↓
사람 학습 / 질문 / 발송 승인
  ↓
PDF/Gmail 발행
  ↓
Drive/Notion/로컬 아카이브
  ↓
지난 가설과 실제 결과 비교
```

이 구조는 초기 MVP에는 좋지만 한계도 있다.

### 한계 1. Google Sheets는 데이터베이스가 아니다

Google Sheets는 사람이 보기 쉽고 수정하기 쉽다. 하지만 데이터가 많아지고 자동화가 늘어나면 실수도 늘어난다.

예시:

- 셀을 잘못 지움
- 수식을 실수로 덮어씀
- 열 이름을 바꿔서 Apps Script가 깨짐
- 날짜 형식이 바뀌어 계산이 틀림

대응:

```text
초기에는 Sheets를 중앙 작업공간으로 쓰되,
중요한 결과물은 CSV와 Markdown으로 로컬에도 백업한다.
```

### 한계 2. Apps Script는 편하지만 복잡한 데이터 수집에는 약하다

Google Apps Script는 Gmail, Drive, Sheets와 연결하기 좋다. 하지만 복잡한 웹 데이터 수집, 에러 처리, 대량 데이터 처리에는 Python보다 불편할 수 있다.

대응:

```text
Apps Script = Sheets 계산, 문서 생성, 승인 전 발송 차단/상태 기록 흐름
Python = 데이터 수집, 데이터 검증, CSV 백업, 출처 비교
```

처음부터 모든 것을 Python으로 옮기지는 않는다. 사용자가 보기 쉬운 중심 작업공간은 Google Sheets로 유지하고, 데이터 수집이 복잡해질 때 Python을 붙인다.

### 한계 3. AI 가설은 그럴듯하지만 틀릴 수 있다

AI는 데이터를 보고 논리적인 문장을 만들 수 있지만, 그 문장이 항상 맞는 것은 아니다. 특히 주식시장은 실적, 금리, 심리, 정책, 수급이 동시에 움직인다.

대응:

```text
AI 가설은 예측이 아니라 학습용 가설로 기록한다.
각 가설에는 근거 지표, 해석 이유, 한계, 다음 확인 항목을 반드시 붙인다.
다음 주와 월간 리뷰에서 실제 결과와 비교한다.
```

### 한계 4. 점수 모델이 복잡해지면 설명력이 떨어진다

점수 항목이 많아지면 그럴듯해 보이지만, 왜 점수가 나왔는지 설명하기 어려워진다.

대응:

```text
점수보다 reasoning_explanation을 더 중요하게 둔다.
8.4점이라는 숫자보다 "왜 8.4점인지"를 먼저 쓴다.
```

### 한계 5. 무료 데이터는 안정성이 낮다

무료 데이터는 매우 유용하지만 안정성이 보장되지 않는다. 사이트 구조가 바뀌거나, 데이터 제공 방식이 바뀌거나, 일부 종목 데이터가 빠질 수 있다.

대응:

```text
무료 데이터는 초안용으로 사용한다.
중요한 내용은 공식 공시, 기업 IR, 배당 발표로 재확인한다.
출처가 약하면 data_confidence를 낮추고 uncertainty_level을 높인다.
```

## 6. 대안과 발전 방향

모든 발전 단계에는 승인 게이트가 있다.

```text
AI 품질 자가검증
  ↓
자동화 발전 제안서 작성
  ↓
사용자 승인 / 보류 / 거절
  ↓
승인된 변경만 적용
  ↓
적용 후 결과 복기
```

AI가 품질이 충분히 좋아졌다고 판단해도 바로 다음 단계로 넘어가지 않는다. AI는 이유와 근거를 설명하고, 사용자의 승인을 받은 뒤에만 발전 단계를 적용한다.

자세한 기준은 `automation/automation-evolution-approval-gate.md`를 따른다.

### 단계 1. 로컬 Markdown + Google Sheets + AI 초안

현재 단계다.

역할:

- Google Sheets: 점수 입력, 후보 선정, 가설 추적
- Markdown: 템플릿, 리포트, 설계 기록
- AI: 점수 계산 보조, 가설 생성, 초보자 설명, 한계 작성
- 사람: 학습, 질문, 발송 승인

장점:

- 쉽다.
- 비용이 거의 없다.
- 수정이 빠르다.
- 사용자가 데이터와 설명을 한눈에 볼 수 있다.

단점:

- 데이터 수집 자동화는 아직 약하다.
- 시트 열 이름이나 수식이 바뀌면 자동화가 깨질 수 있다.

적합한 시점:

```text
지금부터 4주 AI 자동 초안 운영까지
```

다음 단계로 넘어가기 전 승인 조건:

```text
AI가 Google Sheets 구조 반영 제안서를 만들고,
사용자가 승인해야 실제 시트 구조 업데이트나 Code.gs 작성으로 넘어간다.
```

### 단계 2. Google Sheets 수식 + Apps Script 리포트 생성

첫 자동화 강화 단계다.

자동화할 것:

- SSMK 총점 계산
- 관찰 등급 자동 변환
- 지난주 점수 불러오기
- 점수 변화 계산
- 후보 3개 자동 정렬
- AI 가설 입력용 데이터 묶음 생성
- Markdown 또는 Google Docs 초안 생성

사람 승인으로 남길 것:

- 외부 이메일 발송
- 가족/지인 공유
- 면책 문구 포함 여부 확인

적합한 시점:

```text
샘플 CSV 구조가 Google Sheets에 반영된 직후
```

다음 단계로 넘어가기 전 승인 조건:

```text
AI 가설 6단 구조가 반복적으로 유지되고,
사용자 수정량이 줄었다는 근거를 AI가 제시한 뒤,
사용자가 승인해야 리포트 초안 생성 자동화를 강화한다.
```

### 단계 3. Python 데이터 수집 보조

Apps Script만으로 부족해질 때 추가한다.

Python이 맡기 좋은 작업:

- 공시 데이터 다운로드
- CSV 백업
- 여러 출처 데이터 비교
- 데이터 누락 검사
- 보고서용 표 생성
- 지난 가설과 실제 결과 비교용 스냅샷 생성

주의:

```text
Python은 강력하지만 관리할 것이 늘어난다.
처음부터 모든 자동화를 Python으로 만들기보다,
깨지기 쉬운 데이터 수집과 검증부터 맡긴다.
```

다음 단계로 넘어가기 전 승인 조건:

```text
특정 데이터 출처가 3~4주 이상 안정적으로 사용됐고,
수동 확인 부담이 반복된다는 근거가 있을 때만 AI가 Python 도입을 제안한다.
사용자가 승인하기 전에는 Python 자동 수집을 적용하지 않는다.
```

### 단계 4. 로컬 데이터 저장소 추가

리포트가 2~3개월 쌓이면 데이터 이력이 중요해진다. 이때는 단순 Sheets만으로 부족할 수 있다.

대안:

- CSV 백업
- SQLite
- DuckDB

초보자에게 가장 쉬운 순서:

```text
CSV 백업 → SQLite → DuckDB
```

처음에는 CSV 백업만으로 충분하다.

### 단계 5. 대시보드 또는 웹 아카이브

리포트가 쌓이면 나중에 이런 기능을 만들 수 있다.

- 종목별 점수 추이
- 산업별 월간 흐름
- 리스크 플래그 이력
- 지난 AI 가설의 적중/오차 기록
- 어떤 지표가 실제 결과 설명에 도움이 되었는지 기록

이 단계는 급하지 않다.

```text
먼저 좋은 기록을 쌓고,
그 다음에 보기 좋은 화면을 만든다.
```

## 7. 자동화 우선순위

가장 먼저 자동화해도 되는 것:

1. SSMK 총점 계산
2. 관찰 등급 변환
3. 지난주 점수 불러오기
4. 점수 변화 계산
5. 후보 3개 정렬
6. 리포트 목차에 맞춘 Markdown 초안 생성

AI가 적극적으로 맡을 것:

1. AI 가설 3개 생성
2. 근거 지표/데이터 설명
3. 왜 그렇게 해석했는지 설명
4. 초보자 레슨 작성
5. 한계와 다음 확인 항목 작성
6. 지난 가설과 실제 결과 비교 초안 작성

조심해서 자동화할 것:

1. 배당 데이터 수집
2. 실적 일정 수집
3. 뉴스 후보 수집
4. 밸류에이션 지표 수집
5. ADR/OTC 종목 데이터 수집

사람 승인으로 남길 것:

1. 외부 이메일 발송
2. 가족/지인 공유
3. 문장이 투자 추천처럼 읽히는지 최종 확인

## 8. 발행 안전장치

자동화가 들어가기 전부터 아래 안전장치를 문서와 시트에 넣어야 한다.

| 안전장치 | 이유 |
|---|---|
| generation_status | 준비, 초안 생성, 사용자 확인 필요, 승인 상태를 구분 |
| data_confidence | 출처가 약한 데이터가 그대로 발행되는 것을 방지 |
| uncertainty_level | 점수는 높지만 추론이 불안정한 경우를 표시 |
| source_links | 어떤 자료를 근거로 썼는지 확인 |
| risk_flag | 점수보다 리스크를 먼저 보게 함 |
| hypothesis_summary | AI 가설을 한 줄로 기록 |
| evidence_metrics | 가설의 근거 지표를 기록 |
| reasoning_explanation | 왜 그렇게 해석했는지 기록 |
| beginner_lesson | 사용자가 배운 점을 남김 |
| limitations | 현재 추론의 한계를 남김 |
| next_check | 다음에 확인할 항목을 남김 |
| human_approved | 사람이 발송을 승인했는지 표시 |
| generated_at | 자동 생성 시간이 언제인지 기록 |
| automation_stage | 현재 자동화 발전 단계 |
| stage_proposal_status | 발전 제안 상태 |
| change_approval_status | 중요한 변경 승인 상태 |

초보자용 예시:

```text
AI는 리포트의 초안 작성자이자 과외 선생님입니다.
하지만 가족과 지인에게 보내는 발송 버튼은 사용자가 눌러야 합니다.
```

## 9. 권장 다음 작업

다음 단계는 자동화 코드를 바로 길게 쓰는 것이 아니라, 자동화가 읽고 쓸 데이터 구조를 먼저 고정하는 것이다.

1. Google Sheets 열 구조를 `docs/score-model-v1.md`와 `data/sample-score-table.csv` 기준으로 업데이트한다.
2. `weekly_scores`에 가설 관련 컬럼을 추가한다.
3. `hypothesis_reviews` 또는 `lesson_learn` 탭을 새로 설계한다.
4. `automation_stage_reviews`와 `change_approval_log` 탭을 설계한다.
5. AI 자동 리포트 생성 프롬프트를 만든다.
6. Apps Script가 Markdown 초안을 만들 수 있는 구조를 설계한다.
7. 이메일은 초안 생성까지 자동화하고, 실제 발송은 승인 후 진행한다.

가장 중요한 원칙:

```text
AI는 판단을 대신하는 신호등이 아니다.
AI는 데이터를 보고 가설을 만들고, 그 가설의 이유를 설명하는 학습 파트너다.
가설은 실제 결과와 비교되면서 더 나은 질문으로 발전해야 한다.
```
