1. 운영 방식 선택지도

세 가지 방식은 서로 경쟁 관계가 아니라 역할이 다릅니다. 지금은 Codex 예약이 켜져 있고, 안정성을 더 높이려면 Apps Script 자체 트리거를 추가하는 방식이 좋습니다.

A. 지금 반영된 방식: Codex 예약 자동화

Codex가 매주 정해진 시간에 깨어나서 시트와 문서를 확인하고 초안/로그 작업을 수행합니다.

  • 사용자 액션이 가장 적습니다.
  • Codex가 Google Sheets/Docs 커넥터를 이용해 작업합니다.
  • Apps Script 직접 실행 도구가 없으면 대체 실행 경로를 탑니다.
추천 용도: 현재 바로 운영 시작

B. 추천 방식: Apps Script 자체 트리거

Google Apps Script가 자기 안의 시간 예약 기능으로 `runWeeklyLabFullCycle()`을 직접 실행합니다.

  • Google 내부에서 직접 실행되므로 가장 단순합니다.
  • Apps Script 화면에서 트리거를 추가합니다.
  • 첫 설정 때 권한 승인 화면을 한 번 거칩니다.
추천 용도: 정기 운영의 안정성 강화

C. 추가 개발 방식: Apps Script 실행 MCP

Codex에 “Apps Script 실행 버튼 대행 도구”를 붙여서 함수 실행을 직접 요청할 수 있게 만듭니다.

  • Google Cloud와 OAuth 설정이 필요합니다.
  • 설정은 가장 복잡하지만 제어력은 가장 큽니다.
  • 수동 강제 실행과 디버깅에 특히 좋습니다.
추천 용도: 나중에 고급 제어가 필요할 때

5. 설정값을 바꾸는 위치

“화요일 오전 실행” 같은 값은 한 곳만 바꾸면 되는 값이 아닙니다. 어떤 실행 방식의 시간을 바꾸는지 먼저 구분해야 합니다.

Codex 예약 시간 바꾸기

어디서?
Codex 앱의 예약 자동화 카드, 또는 이 채팅에서 “SSMK 예약을 매주 화요일 오전 9시로 바꿔줘”라고 요청
무엇을 바꾸나?
실제 Codex 예약 실행 시간
예시
“매주 화요일 오전 8시” → “매주 화요일 오전 9시”
주의
이 변경은 운영 흐름 변경이므로 앞으로도 승인 기록을 남기는 것이 좋습니다.

Google Sheets 정책값 바꾸기

어디서?
Google Sheets → automation_schedules 탭 또는 SSMK Control Center
무엇을 바꾸나?
“이 자동화를 운영 대상으로 볼지”에 대한 정책 표시
예시
tuesday_weekly_report의 enabled 값을 ON/OFF로 변경
주의
이 값만 바꿔도 Codex 예약 시간이 자동으로 바뀌지는 않습니다.

Apps Script 트리거 시간 바꾸기

어디서?
Google Sheets → 확장 프로그램 → Apps Script → 왼쪽 트리거 메뉴
무엇을 바꾸나?
Google Apps Script 자체 예약 실행 시간
예시
기존 트리거를 수정하거나 삭제 후 새 트리거를 추가
주의
Codex 예약과 시간이 겹치면 같은 작업이 두 번 실행될 수 있습니다.

이메일 발송 켜기

어디서?
SSMK Control Center에서는 먼저 이메일용 HTML 최종본 만들기로 검토용 HTML 파일만 만듭니다.
무엇이 필요한가?
실제 발송은 report_runs.generation_status = 승인 상태일 때만 sendApprovedReport(reportId)로 가능합니다.
현재 값
user_preferences.email_auto_send = OFF
주의
HTML 최종본 생성과 실제 이메일 발송은 다른 단계입니다. HTML 파일을 만들어도 메일은 자동으로 가지 않습니다.

6. 실행 후 확인 체크리스트

자동화가 잘 돌았는지는 “메일이 왔는지”가 아니라 Google Sheets 로그가 쌓였는지로 확인합니다.

1
automation_run_log 확인 새 run_id, started_at, ended_at, status가 있는지 봅니다.
2
automation_step_log 확인 collect_weekly_back_data, build_weekly_scores, create_visualization_queue, create_email_html_final_draft 같은 단계가 성공했는지 봅니다.
3
report_runs 확인 새 report_id와 Google Docs 링크가 생겼는지 봅니다.
4
qa_review_log 확인 overall_status와 recommended_next_action을 읽습니다.
5
error_log 확인 비어 있으면 좋고, 값이 있으면 error_message와 recovery_action을 봅니다.
6
visualization_queue 확인 자동 시각화 행이 report_id 기준으로 생겼는지 봅니다.
7
이메일 미발송 확인 report_runs의 sent_at이 비어 있고, email_auto_send가 OFF인지 확인합니다.
“처음부터 다시 실행”은 오늘 날짜의 작업 재료인 market_data, news_events, weekly_scores, visualization_queue, hypothesis_reviews를 지우고 새로 만듭니다. 반대로 실행 이력과 문서 이력인 automation_run_log, report_runs, report_versions, qa_review_log는 나중에 원인을 추적할 수 있도록 남깁니다.
