# 자동화 발전 승인 게이트

작성일: 2026-04-22

## 1. 목적

이 문서는 SSMK 투자 관찰노트의 자동화 수준을 언제 높일지, 어떤 변경은 반드시 사용자 승인을 받아야 하는지 정리한다.

핵심 원칙:

```text
AI는 자동화 발전이 필요하다고 판단할 수 있다.
하지만 AI가 스스로 발전 단계를 적용하지 않는다.
AI는 근거를 정리해 사용자에게 제안하고, 사용자가 승인한 뒤에만 적용한다.
```

초보자용으로 말하면, AI는 자동차의 계기판과 내비게이션 역할을 한다. "이제 고속도로에 올라가도 될 것 같습니다"라고 알려줄 수는 있지만, 실제로 차선을 바꾸는 결정은 운전자인 사용자가 한다.

---

## 2. 왜 승인 게이트가 필요한가

자동화는 편해질수록 위험도 함께 커진다.

예를 들어 아래 변경은 겉보기에는 효율 개선처럼 보이지만, 콘텐츠의 성격을 바꿀 수 있다.

- 뉴스 수집 출처를 바꾸는 것
- SSMK 점수 비중을 바꾸는 것
- AI 가설 작성 방식을 바꾸는 것
- 이메일 발송 대상을 늘리는 것
- PDF/Gmail 발송을 더 자동화하는 것
- 무료 데이터 대신 유료/외부 API를 도입하는 것

이런 변경은 사용자가 놓치기 쉽다. 그래서 AI가 먼저 감지하고, "이제 다음 단계로 가도 될 것 같습니다" 또는 "중요 변경이 필요해 보입니다"라고 알려주는 프로세스가 필요하다.

---

## 3. 매주 AI가 자가 검증할 항목

주간 리포트가 생성될 때마다 AI는 아래 항목을 스스로 점검한다.

| 점검 항목 | 확인 질문 | 좋은 상태 |
|---|---|---|
| 가설 구조 완성도 | 모든 AI 가설이 6단 구조인가? | 3개 가설 모두 완성 |
| 근거 연결성 | 근거 지표가 가설과 실제로 연결되는가? | 근거와 해석 이유가 분리되어 있음 |
| 초보자 이해도 | 지표 의미를 배울 수 있는 설명인가? | 비유 또는 쉬운 설명 포함 |
| 한계 고지 | 모르는 점을 솔직히 적었는가? | 데이터 신뢰도 낮음/불확실성 높음에 한계 표시 |
| 투자 추천 방지 | 매수/매도처럼 읽히는 표현이 없는가? | 행동 유도 표현 없음 |
| 데이터 출처 안정성 | 반복 사용 가능한 출처인가? | 공식/안정 출처 우선 |
| 복기 가능성 | 다음에 실제 결과와 비교 가능한가? | next_check가 구체적 |
| 사용자 수정량 | 사용자가 많이 고쳐야 했는가? | 반복 수정 항목이 줄어듦 |

## 4. 월간 AI 품질 리뷰

월간 리뷰에서는 주간보다 더 넓게 본다.

AI는 아래 질문에 답해야 한다.

```text
1. 이번 달 AI 가설 중 실제 결과와 비교 가능한 가설은 몇 개인가?
2. 가설의 근거 지표는 실제 결과를 설명하는 데 도움이 되었는가?
3. 반복적으로 틀리거나 부족했던 해석 패턴은 무엇인가?
4. 사용자가 반복해서 질문한 지표나 개념은 무엇인가?
5. 데이터 출처 중 자주 깨지거나 불안정한 것은 무엇인가?
6. 다음 자동화 단계로 넘어가도 될 만큼 품질이 안정됐는가?
```

## 5. 발전 단계 제안 기준

AI는 아래 조건이 충족되면 사용자에게 발전 단계 제안을 올린다.

### 5-1. Google Sheets 구조 안정화 제안

제안 조건:

- 샘플 데이터가 주간 리포트 템플릿을 채우기에 충분함
- 빠진 컬럼 때문에 리포트 작성이 막히지 않음
- 가설 복기용 컬럼이 실제로 사용 가능함

제안 문장 예시:

```text
최근 리포트 구조를 보면 `weekly_scores`와 `hypothesis_reviews` 컬럼이 안정적으로 사용되고 있습니다.
이제 실제 Google Sheets에 해당 구조를 반영해도 될 단계로 보입니다.
승인하면 Google Sheets 컬럼 반영 절차로 넘어가겠습니다.
```

### 5-2. AI 리포트 초안 자동화 강화 제안

제안 조건:

- 2~4회 연속 리포트에서 AI 가설 6단 구조가 유지됨
- 사용자 수정이 주로 문장 다듬기 수준임
- 투자 추천처럼 읽히는 표현이 반복적으로 나오지 않음

제안 문장 예시:

```text
최근 4회 리포트에서 AI 가설 구조가 안정적으로 유지됐고, 수정 포인트가 문장 표현 중심으로 줄었습니다.
이제 주간 리포트 초안 생성을 더 자동화하는 단계로 넘어가도 좋겠습니다.
승인하면 `Code.gs` 초안 작성 또는 자동 초안 생성 테스트를 진행하겠습니다.
```

### 5-3. 데이터 수집 자동화 확대 제안

제안 조건:

- 특정 데이터 출처가 3~4주 연속 안정적으로 사용됨
- 수동 확인 시간이 반복적으로 많이 듦
- 데이터의 의미와 한계가 문서화되어 있음

제안 문장 예시:

```text
배당 데이터와 실적 일정은 4주 연속 같은 방식으로 확인됐고, 매주 반복 입력 부담이 큽니다.
이 두 항목은 자동 수집 후보로 올려도 될 것 같습니다.
다만 ADR/OTC 종목은 데이터 신뢰도가 낮아 제외하는 안을 추천합니다.
이 방향으로 진행해도 될까요?
```

### 5-4. 이메일/PDF 자동화 확대 제안

제안 조건:

- 리포트 품질 체크리스트가 안정적으로 통과됨
- 면책 문구 누락이 없음
- 발송 대상과 승인 상태가 명확히 관리됨

항상 유지할 원칙:

```text
외부 이메일 발송은 사용자의 명시 승인 없이는 자동 실행하지 않는다.
```

## 6. 반드시 사용자 승인이 필요한 변경

아래 변경은 AI가 필요하다고 판단해도 반드시 사용자에게 먼저 알리고 승인을 받아야 한다.

| 변경 유형 | 왜 중요한가 |
|---|---|
| SSMK 점수 비중 변경 | 점수 해석이 달라짐 |
| 관찰 등급 기준 변경 | 높음/중간/낮음의 의미가 바뀜 |
| 산업 구분 변경 | 월간 비교 기준이 바뀜 |
| 워치리스트 종목 추가/삭제 | 관찰 범위가 바뀜 |
| 데이터 출처 변경 | 데이터 신뢰도와 비용이 바뀜 |
| 유료 API 또는 유료 도구 도입 | 비용 발생 가능 |
| AI 프롬프트 구조 변경 | 리포트 문체와 판단 방식이 바뀜 |
| 이메일 수신자 변경 | 외부 공유 범위가 바뀜 |
| 자동 발송 수준 변경 | 콘텐츠 리스크가 커짐 |
| 투자 추천처럼 보일 수 있는 표현 정책 변경 | 프로젝트 철학 훼손 가능 |

## 7. 사용자에게 올릴 제안서 형식

AI는 중요한 변경이나 자동화 발전이 필요하다고 판단하면 아래 형식으로 제안한다.

```text
제안 제목:
[예: 데이터 수집 자동화 확대 제안]

현재 상태:
[최근 몇 주 동안 관찰된 상황]

AI가 이렇게 판단한 근거:
[품질 지표, 반복 작업, 오류, 사용자 수정량, 데이터 안정성]

추천하는 변경:
[구체적으로 무엇을 바꿀지]

기대 효과:
[시간 절약, 품질 개선, 복기 정확도 개선 등]

위험과 한계:
[깨질 수 있는 부분, 비용, 오해 가능성]

되돌리는 방법:
[문제가 생기면 이전 상태로 돌아가는 방법]

사용자 승인 요청:
이 방향으로 진행해도 될까요?
```

## 8. 승인 상태값

Google Sheets 또는 로컬 기록에서는 아래 상태값을 사용한다.

| 상태 | 의미 |
|---|---|
| proposed | AI가 제안했지만 아직 승인 전 |
| approved | 사용자가 승인함 |
| rejected | 사용자가 거절함 |
| postponed | 나중에 다시 보기로 함 |
| applied | 승인된 변경을 적용함 |
| rolled_back | 적용 후 문제가 있어 되돌림 |

risk
rollback_plan

approved_at
