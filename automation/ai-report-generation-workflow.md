# AI 자동 리포트 생성 흐름

작성일: 2026-04-22

## 1. 목적

이 문서는 SSMK 투자 관찰노트를 매주 이메일 발행 가능한 형태로 자동 생성하기 위한 흐름을 정리한다.

핵심 원칙:

```text
AI는 결론만 쓰지 않는다.
AI는 가설, 근거 지표, 해석 이유, 초보자 레슨, 한계, 다음 확인을 함께 쓴다.
```

이 구조가 중요한 이유는 사용자가 주식 초보자이기 때문이다. 리포트는 "이 종목이 좋아 보입니다"라고 말하는 콘텐츠가 아니라, "이 지표를 보면 이런 질문을 할 수 있습니다"라고 알려주는 학습 도구여야 한다.

## 2. 전체 자동화 흐름

```text
Google Sheets 데이터 준비
  ↓
AI 입력용 데이터 묶음 생성
  ↓
벡터: 데이터 검증
  ↓
루미: 점수/가설/설명 기반 리포트 초안 생성
  ↓
Markdown 주간 리포트 초안 생성
  ↓
세이지: 품질/추천화 위험 검토
  ↓
파일럿: 승인/발송/복기 상태 확인
  ↓
노바: 자동화 발전 제안 여부 판단
  ↓
사람이 학습자 입장에서 읽고 질문
  ↓
문장 수정 및 발송 승인
  ↓
PDF 또는 이메일 본문 생성
  ↓
Gmail 발송
  ↓
Drive/로컬/Notion 아카이브
  ↓
hypothesis_reviews에 복기 예약
  ↓
콘텐츠 품질 자가검증
  ↓
필요하면 자동화 발전 제안서 작성
  ↓
사용자 승인 후에만 다음 단계 적용
```

## 3. 주간 운영 흐름

### 월요일: 데이터 준비

AI 또는 자동화가 준비할 자료:

- 주가 변화
- 지난주 SSMK 점수
- 최신 실적/IR/공시 링크
- 배당 관련 데이터
- 주요 뉴스 후보
- 산업별 주요 이벤트
- 실적 발표 예정일

월요일의 목표:

```text
리포트를 쓰는 것이 아니라, 화요일에 AI가 해석할 재료를 모으는 날입니다.
```

### 화요일: AI 가설과 리포트 초안 생성

AI가 할 일:

1. 벡터가 데이터 출처, 최신성, 신뢰도, 불확실성을 확인한다.
2. 루미가 종목별 SSMK 총점, 관찰 등급, 가설, 초보자 설명을 바탕으로 초안을 만든다.
3. 세이지가 가설 6단 구조, 추천화 표현, 한계 누락을 검토한다.
4. 파일럿이 report_runs, hypothesis_reviews, 복기 예약 상태를 확인한다.
5. 노바가 자동화 발전 제안이 필요한지 판단한다.
6. 최종 상태를 `발행 가능 / 사용자 확인 필요 / 발행 보류 권장` 중 하나로 표시한다.

화요일의 핵심 산출물:

```text
reports/YYYY-MM-DD-weekly-report-draft.md
```

추가로 AI가 점검할 것:

```text
이번 리포트가 다음 자동화 단계로 넘어갈 만큼 안정적인가?
아직 사용자 수정이 많이 필요한가?
데이터 출처가 반복적으로 안정적인가?
가설 6단 구조가 무너지지 않았는가?
```

### 수요일: 학습, 질문, 승인, 발송

사람이 할 일:

1. 리포트를 읽는다.
2. 이해되지 않는 지표를 AI에게 질문한다.
3. 고치고 싶은 부분은 `SSMK Control Center`의 재작업 요청으로 남긴다.
4. 너무 단정적인 문장을 수정한다.
5. 면책 문구가 들어갔는지 확인한다.
6. 발송을 승인한다.

재작업 요청 흐름:

```text
SSMK Control Center에서 리포트 ID를 입력한다
→ 특정 섹션 또는 전체 리포트 중 하나를 고른다
→ 재작업 유형과 요청사항을 적는다
→ revision_requests에 status=requested로 저장된다
→ 다음 재작업 처리 단계에서 해당 요청을 읽고 반영한다
```

섹션/버전 기록 흐름:

```text
리포트 초안이나 재작업 결과가 만들어진다
→ upsertReportSection_()으로 report_sections에 섹션 상태를 남긴다
→ createReportVersion_()으로 report_versions에 v1, v2 같은 버전 이력을 남긴다
→ source_request_id로 어떤 재작업 요청 때문에 바뀌었는지 연결한다
```

주의:

```text
재작업 요청을 저장하는 것은 발송 승인이 아니다.
요청이 접수되어도 이메일 발송은 계속 사용자 승인 전까지 막혀 있다.
```

발송 승인 후 자동화가 할 일:

- PDF 생성
- Gmail 발송
- Google Drive 보관
- 로컬 `reports` 폴더 보관
- `report_runs` 상태 업데이트

## 4. AI 가설 생성 프롬프트 구조

AI에게 가설을 만들게 할 때는 아래 구조를 고정한다.

```text
너는 SSMK 투자 관찰노트의 AI 분석가이자 초보자용 투자 공부 선생님이다.
아래 데이터는 투자 추천을 만들기 위한 것이 아니라, 학습용 관찰 가설을 만들기 위한 자료다.

각 가설은 반드시 아래 구조로 작성해라.

1. 가설 요약
2. 근거 지표/데이터
3. 이 지표/데이터를 보고 왜 그렇게 예상했나
4. 초보자가 배울 포인트
5. 현재 한계
6. 다음 확인

주의:
- 매수/매도 추천처럼 쓰지 마라.
- 점수가 높다고 좋은 투자 대상이라고 말하지 마라.
- 데이터 신뢰도가 낮으면 반드시 한계를 써라.
- 최신 실적 확인 전이면 확정 표현을 쓰지 마라.
- 지표의 의미를 초보자도 이해할 수 있게 예시나 비유로 설명해라.
```

## 5. 가설 품질 체크리스트

AI 가설은 아래 질문을 통과해야 한다.

- 가설이 한 문장으로 요약되는가?
- 근거 지표가 실제로 가설과 연결되는가?
- 왜 그 지표를 보고 그런 해석을 했는지 설명했는가?
- 초보자가 그 지표의 의미를 배울 수 있는가?
- 현재 모르는 점을 솔직하게 적었는가?
- 다음에 어떤 데이터로 확인할지 구체적인가?

나쁜 예:

```text
NVDA는 AI 수요가 좋아서 계속 긍정적입니다.
```

좋은 예:

```text
가설 요약:
NVDA는 데이터센터 매출이 유지되면 AI 인프라 수요가 아직 강하다는 관찰 가설을 세울 수 있습니다.

근거 지표/데이터:
데이터센터 매출 성장률, GPU 수요 관련 가이던스, 매출총이익률.

왜 이렇게 해석했나:
AI 뉴스가 많아도 실제 수요는 기업들이 GPU와 데이터센터 장비에 돈을 쓰는지에서 확인됩니다. 데이터센터 매출이 유지되면 AI 기대가 실제 구매로 이어지고 있다는 신호일 수 있습니다.

초보자 레슨:
성장 테마는 뉴스보다 매출과 마진으로 확인해야 합니다.

한계:
이미 높은 기대가 가격에 반영되어 있으면 좋은 실적에도 주가 반응이 약할 수 있습니다.

다음 확인:
다음 실적에서 데이터센터 매출 성장률과 다음 분기 가이던스를 확인합니다.
```

## 6. 자동화 발전 자가검증

주간 리포트 초안 생성 후 AI는 아래를 평가한다.

| 항목 | 질문 |
|---|---|
| 구조 안정성 | AI 가설 3개가 모두 6단 구조를 지켰는가? |
| 설명 품질 | 초보자가 지표 의미를 이해할 수 있는가? |
| 데이터 안정성 | 출처가 반복적으로 안정적인가? |
| 수정 부담 | 사용자가 반복적으로 많이 고쳐야 하는가? |
| 복기 가능성 | 다음 주/4주 후 실제 결과와 비교 가능한가? |
| 리스크 통제 | 투자 추천처럼 읽히는 표현이 없는가? |

AI가 다음 단계로 넘어가도 된다고 판단하면 아래 형식의 제안서를 만든다.

```text
제안 제목
현재 상태
AI가 이렇게 판단한 근거
추천하는 변경
기대 효과
위험과 한계
되돌리는 방법
사용자 승인 요청
```

중요:

```text
AI는 제안할 수 있지만 승인 없이 적용하지 않는다.
```

## 6-1. 에이전트 리뷰 보드

주간 리포트 초안은 아래 순서로 리뷰한다.

| 순서 | 에이전트 | 역할 | 실패 시 상태 |
|---:|---|---|---|
| 1 | 벡터 | 데이터 검증 | 사용자 확인 필요 또는 발행 보류 권장 |

| 4 | 파일럿 | 승인/발송/복기 흐름 확인 | 사용자 확인 필요 |
| 5 | 노바 | 자동화 발전 제안 판단 | 제안서 작성, 적용은 보류 |

최종 상태:

```text
발행 가능
사용자 확인 필요
발행 보류 권장
```

자세한 역할 규칙은 `automation/ai-agent-roles-and-review-board.md`를 따른다.

## 8. 발송 승인 규칙

리포트는 아래 조건을 만족해야 발송 가능하다.

- 모든 AI 가설에 근거 지표가 있다.
- 모든 AI 가설에 해석 이유가 있다.
- 모든 AI 가설에 초보자 레슨이 있다.
- 데이터 신뢰도가 낮은 내용에는 한계가 표시되어 있다.
- 투자 추천처럼 읽히는 문장이 없다.
- 면책 문구가 상단과 하단에 있다.
- `report_runs.review_status`가 `승인`이다.

자동화 발전 적용은 아래 조건도 필요하다.

```text
automation_stage_reviews.approval_status = approved
change_approval_log.approval_status = approved
```

단, 이 조건은 자동화 발전 적용에만 사용한다. 리포트 발송 승인은 별도로 유지한다.

## 10. Weekly Lab 워크플로 초안

현재 Weekly Lab 쪽에서는 아래 2개 함수를 새 기본 초안으로 본다.

```text
runWeeklyLabWorkflow(issueDate)
createWeeklyLabPromptDoc_(issueDate, runId)
```

각 함수 역할:

```text
runWeeklyLabWorkflow():
- run_id를 만든다
- 시트 구조를 조용히 점검한다
- Weekly Lab 입력 프롬프트 문서를 만든다
- 가설 복기 예약을 만든다
- 에이전트 리뷰 보드를 실행한다
- report_runs와 automation_run_log 상태를 남긴다
- 가능하면 operator QA 리뷰도 남긴다

createWeeklyLabPromptDoc_():
- 최종 리포트가 아니라 Codex 자동화가 읽을 입력 문서를 만든다
- Google Docs 문서에 run_id, report_id, issue_date, 기준 파일 경로, 입력 데이터 JSON을 넣는다
- report_sections와 report_versions에도 최소 기록을 남긴다
```

초보자용 설명:

```text
이 문서는 아직 "리포트를 바로 완성하는 버튼"이 아니다.
먼저 자동화가 읽을 입력 문서를 만들고,
그다음 로그와 검토 결과를 남기는 "작업 준비 버튼"에 가깝다.
```

호환성 규칙:

```text
기존 runWeeklyDraftPrepWorkflow()는 당장 지우지 않는다.
이전 흐름을 참고하거나 비교할 수 있게 하위 호환용으로 남겨 둔다.
메뉴 기본 실행 항목 교체는 다음 단계에서 따로 정리한다.
```
