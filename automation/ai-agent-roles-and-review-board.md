# AI 에이전트 역할 분리와 리뷰 보드 설계

작성일: 2026-04-22

## 1. 목적

이 문서는 SSMK 투자 관찰노트에서 AI 역할을 하나의 거대한 AI가 아니라, 서로 다른 책임을 가진 여러 역할 에이전트로 나누어 운영하기 위한 설계다.

핵심 목표:

```text
콘텐츠를 잘 만드는 AI와,
콘텐츠를 냉정하게 검증하는 AI와,
데이터만 보는 AI와,
프로세스를 감시하는 AI를 분리한다.
```

이렇게 나누는 이유는 단순하다. 리포트를 만든 AI가 자기 리포트를 스스로 후하게 평가하면 품질 저하, 데이터 이상, 복기 누락, 추천화, 승인 피로를 놓칠 수 있다. 따라서 역할별로 관점을 분리하고, 서로의 결과를 냉정하게 검토하게 한다.

---

## 2. 설계 판단

이 아이디어는 SSMK 프로젝트에 적합하다.

이유:

- 프로젝트의 목적이 투자 추천이 아니라 학습형 관찰 기록이다.
- 초보자인 사용자가 모든 오류와 타이밍을 직접 감지하기 어렵다.
- AI가 콘텐츠 생산과 품질 검증을 동시에 맡으면 자기검열이 약해질 수 있다.
- 데이터 오류, 표현 위험, 복기 누락, 자동화 타이밍은 서로 다른 전문 관점이 필요하다.

다만 처음부터 에이전트를 너무 많이 나누면 운영 복잡도가 커진다. 그래서 초기에는 5개 핵심 역할로 시작한다.

---

## 3. 핵심 에이전트 5개

### 3-1. 루미: 콘텐츠 크리에이터 에이전트

역할:

- 주간 리포트 초안 작성
- AI 가설 3개 작성
- 초보자용 해석 작성
- 레슨&런 작성
- 이메일로 읽기 쉬운 문장 구성

좋은 상태:

```text
초보자가 읽고 "이 지표가 이런 의미였구나"라고 이해할 수 있다.
```

주의할 점:

```text
루미는 문장을 매끄럽게 만들 수 있지만, 그럴듯한 설명으로 위험을 덮을 수 있다.
따라서 루미의 결과물은 반드시 다른 에이전트가 검토한다.
```

### 3-2. 벡터: 데이터 검증 에이전트

역할:

- 데이터 출처 확인
- 오래된 데이터 감지
- 출처 간 수치 차이 감지
- 데이터 신뢰도와 불확실성 평가
- ADR/OTC, 무료 데이터, 뉴스 기반 데이터의 한계 표시

좋은 상태:

```text
리포트에 들어간 숫자와 출처가 최소한 "무엇을 기준으로 한 숫자인지" 설명된다.
```

차단 권한:

```text
벡터가 데이터 오류 또는 신뢰도 낮음을 발견하면 발행 상태를 "사용자 확인 필요" 또는 "발행 보류 권장"으로 올릴 수 있다.
```

### 3-3. 세이지: 편집자/품질 검증 에이전트

역할:

- 리포트가 투자 추천처럼 읽히는지 검토
- AI 가설 6단 구조 검토
- 초보자 설명 품질 검토
- 표현이 과장되거나 단정적인지 검토
- 면책 문구와 발행 전 체크리스트 확인

좋은 상태:

```text
문장이 매끄러운 것보다 안전하고 배우기 좋은지가 먼저 확인된다.
```

차단 권한:

```text
세이지가 추천화 표현, 한계 누락, 초보자 설명 부족을 발견하면 발행 전 수정을 요구한다.
```

### 3-4. 파일럿: 운영/프로세스 감독 에이전트

역할:

- 주간 운영 일정 관리
- 리포트 생성, 승인, 발송 상태 확인
- 복기 예약과 누락 감지
- 승인 피로 감지
- report_runs, hypothesis_reviews 상태 확인

좋은 상태:

```text
리포트가 만들어졌는지뿐 아니라, 복기와 승인 기록까지 흐름이 끊기지 않는다.
```

경고 조건:

```text
복기 대상이 밀리거나, 승인 없이 발송되거나, 사용자가 반복적으로 내용을 보지 않고 승인하는 패턴이 보이면 경고한다.
```

### 3-5. 노바: 자동화 발전/변경 제안 에이전트

역할:

- 자동화 발전 타이밍 판단
- 프로세스 변경 필요성 감지
- 점수 모델, 데이터 출처, 프롬프트, 발송 범위 변경 제안
- 승인 게이트 제안서 작성
- 변경 후 되돌림 계획 확인

좋은 상태:

```text
사용자가 놓칠 수 있는 개선 타이밍을 AI가 먼저 알려준다.
하지만 사용자 승인 없이 변경하지 않는다.
```

차단 권한:

```text
위험과 되돌림 방법이 없는 변경 제안은 진행하지 않는다.
```

## 4. 선택적 에이전트

초기에는 5개로 충분하다. 다만 운영이 쌓이면 아래 역할을 추가할 수 있다.

| 에이전트 | 역할 | 추가 시점 |
|---|---|---|
| 아카이브 | 리포트/데이터/Notion/Drive 보관 상태 확인 | 리포트가 4회 이상 쌓인 뒤 |
| 멘토 | 사용자가 반복해서 헷갈리는 지표를 학습 커리큘럼으로 정리 | 레슨&런이 누적된 뒤 |
| 레드팀 | 일부러 반대 관점에서 가설을 공격 | 가설이 너무 한 방향으로 쏠릴 때 |

---

## 5. 주간 운영 플로우

```text
1. 벡터가 데이터 재료를 확인한다.
2. 루미가 주간 리포트 초안을 만든다.
3. 세이지가 콘텐츠 품질과 추천화 위험을 검토한다.
4. 파일럿이 승인/발송/복기 상태를 확인한다.
5. 노바가 자동화 발전 또는 변경 제안 필요성을 판단한다.
6. 문제가 없으면 사용자에게 발행 승인 요청을 올린다.
7. 문제가 있으면 "사용자 확인 필요" 또는 "발행 보류 권장"으로 표시한다.
```

## 6. 상태 판정

각 리포트는 최종적으로 아래 3개 상태 중 하나를 받는다.

| 상태 | 의미 | 사용자 액션 |
|---|---|---|
| 발행 가능 | 큰 문제 없음 | 읽고 승인 가능 |
| 사용자 확인 필요 | 데이터/표현/가설 중 확인할 부분 있음 | 질문 후 승인 또는 수정 |
| 발행 보류 권장 | 데이터 오류, 추천화, 복기 누락 등 중대 문제 | 수정 후 재검토 |

---

## 7. 에이전트 간 견제 규칙

### 7-1. 루미는 자기 콘텐츠를 최종 승인하지 못한다

콘텐츠를 만든 에이전트는 최종 품질 판정을 하지 않는다.

```text
루미 작성 → 세이지 검토 → 벡터 데이터 확인 → 파일럿 상태 확인
```

### 7-2. 벡터는 문장 품질보다 데이터 진실성을 우선한다

벡터는 문장이 예쁘게 보이는지 신경 쓰지 않는다. 숫자, 출처, 최신성, 기준이 맞는지만 본다.

### 7-3. 세이지는 사용자에게 불편한 판정도 해야 한다

세이지는 리포트가 멋져 보여도 추천처럼 읽히면 수정 요구를 해야 한다.

### 7-4. 파일럿은 일정과 누락을 본다

파일럿은 콘텐츠의 내용보다 프로세스가 끊기지 않았는지 본다.

### 7-5. 노바는 제안만 하고 적용하지 않는다

노바는 다음 단계 제안서를 만들 수 있지만, 사용자 승인 전에는 적용하지 않는다.

---

## 8. 충돌 해결 규칙

에이전트 의견이 충돌하면 아래 우선순위를 따른다.

1. 데이터 오류 또는 출처 불명확성
2. 투자 추천처럼 읽히는 표현
3. 발행 승인/수신자/외부 공유 리스크
4. 복기 누락 또는 가설 검증 불가
5. 자동화 발전 제안
6. 문장 표현과 가독성

예시:

```text
루미: 리포트 문장이 자연스럽고 읽기 좋음
벡터: 최신 실적 데이터가 확인되지 않음
세이지: 일부 표현이 추천처럼 읽힘

결론:
발행 보류 권장. 데이터 확인과 표현 수정 후 재검토.
```

---

## 9. 사용자에게 보여줄 최종 판정 형식

```text
이번 리포트 에이전트 리뷰 결과

루미: 초안 작성 완료
벡터: 데이터 신뢰도 중간, 일부 출처 재확인 필요
세이지: 추천화 표현 2개 수정 필요
파일럿: 복기 예약 정상
노바: 자동화 발전 제안 없음

최종 상태:
사용자 확인 필요

필요 액션:
1. 데이터 출처 1개 확인
2. 추천처럼 읽히는 문장 2개 수정
3. 수정 후 발행 승인 가능
```

## 10. Google Sheets 반영 추천

추가 추천 탭:

```text
agent_review_log
automation_step_log
error_log
bottleneck_log
qa_review_log
```

권장 컬럼:

```text
review_id
issue_date
agent_name
agent_role
review_target
status
finding_summary
risk_level
required_action
blocking
resolved
resolved_at
notes
```

상태값:

```text
pass
warning
block
proposal
```

운영 로그 역할:

| 탭 | 역할 |
|---|---|
| `automation_step_log` | 각 실행 단계가 언제, 어떤 상태로 끝났는지 기록 |
| `error_log` | 오류가 난 단계, 심각도, 원인 추정, 복구 방법 기록 |
| `bottleneck_log` | 느린 단계, 반복 실패, 수동 작업 병목 기록 |
| `qa_review_log` | 실행이 끝난 뒤 오퍼레이터 관점에서 콘텐츠, 데이터, 시각화, 프로세스를 한 줄로 정리하는 최종 점검 기록 |

초보자용 설명:

```text
agent_review_log가 "콘텐츠 검토 결과표"라면,
automation_step_log, error_log, bottleneck_log는 "작업 과정 CCTV"에 가깝다.
qa_review_log는 "마지막 검사표"에 가깝다.
문제가 생겼을 때 어디서 멈췄는지 거꾸로 따라가기 위한 기록이다.
```

## 11. 최종 원칙

```text
에이전트 분리는 자동화를 복잡하게 만들기 위한 것이 아니다.
사용자가 놓칠 수 있는 위험을 서로 다른 관점에서 잡기 위한 장치다.

콘텐츠를 만드는 AI와 검증하는 AI는 분리한다.
데이터를 보는 AI와 문장을 보는 AI도 분리한다.
프로세스 변경은 반드시 사용자 승인 후 적용한다.
```
